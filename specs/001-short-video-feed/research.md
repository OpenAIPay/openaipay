# Phase 0 研究：短视频信息流应用

## 决策 1：在现有 `openaipay` 单仓内新增 `shortvideo` 领域模块，而不是新建独立仓库

**Decision**: 后端沿用当前 `domain / application / infrastructure / adapter-web`
分层结构，在每一层新增 `shortvideo` 命名空间；iOS 端在 `ios-app/OpenAiPay/Features`
下新增 `ShortVideo` 功能目录。

**Rationale**: 用户明确要求“体系结构相同但模块相对独立，便于后续合并”。现有仓库已经
形成稳定的多模块单仓结构、迁移机制、认证体系和运行脚本，直接在此基础上增加模块最符合
后续合并诉求。

**Alternatives considered**:
- 新建独立短视频仓库：模块隔离更强，但会增加后续合并和基础设施复用成本。
- 在现有支付或消息模块中零散扩展：实现速度可能更快，但会放大领域耦合，违背“相对独立”目标。

## 决策 2：第一阶段不新增短视频专用 BFF，iOS 直接调用 `adapter-web` 的短视频 REST API

**Decision**: 短视频浏览与互动接口直接由 Java 后端 `adapter-web` 提供，iOS 端通过现有
`Core/APIClient.swift` 和认证状态能力直接访问；`app-bff` 在第一阶段保持现状。

**Rationale**: 当前仓库里 iOS 端本身已经同时具备访问 BFF 和 Backend 的能力。短视频
场景以登录态、视频流和互动回写为主，不需要复杂的跨服务聚合。直接接后端可以减少重复封装，
也更有利于保持短视频模块在后端内部的边界完整。

**Alternatives considered**:
- 经 `app-bff` 统一代理：适合跨端聚合，但当前阶段会增加一层维护成本。
- iOS 同时混用 BFF 聚合和 Backend 直连：灵活但复杂度高，不利于短视频模块边界清晰。

## 决策 3：视频文件采用 “S3 兼容对象存储 + CDN” 方案，本地开发可用 MinIO 兼容环境

**Decision**: 视频与封面不保存在 MySQL，也不继续沿用当前图片本地文件目录，而是统一放到
S3 兼容对象存储中，通过 CDN 暴露播放地址；本地开发环境可使用 MinIO 或任意 S3 兼容实现。

**Rationale**: 现有后端的 `media` 模块已具备媒资抽象和上传/查询能力，但当前默认配置
更偏图片与本地文件。短视频文件体积更大、分发要求更高，使用 S3 兼容对象存储能保持厂商
无关性，也更符合未来上线和 CDN 分发场景。

**Alternatives considered**:
- 本地文件系统：实现简单，但不利于视频规模扩展与 CDN 分发。
- 强绑定某云厂商视频点播服务：能力强，但会降低环境迁移与本地联调灵活性。

## 决策 4：播放协议优先 HLS，接口层只暴露统一播放地址与媒资元数据

**Decision**: 计划中将短视频播放资源抽象为统一的 `playbackUrl + protocol + mimeType`
输出，生产形态优先 HLS；本地或演示环境允许使用兼容的 MP4 回源文件。

**Rationale**: iOS 上使用 AVPlayer/AVKit 播放 HLS 是成熟路径，后续更容易接入码率自适应
和 CDN 缓存策略。接口层保持统一输出，能减少客户端与存储实现的耦合。

**Alternatives considered**:
- 只支持 MP4 直链：实现简单，但后续扩展到更真实的视频分发能力时改动较大。
- 直接暴露厂商 SDK 播放协议：能力强，但会过早绑定具体基础设施。

## 决策 5：复用现有登录、令牌和用户上下文能力，短视频主流程要求已登录

**Decision**: 短视频功能不新建独立账号体系，继续复用现有 `/api/auth` 登录能力和
`UserSecurityInterceptor` 鉴权链路；进入短视频主流程前必须完成登录。

**Rationale**: 仓库现有认证链路已经稳定，且短视频点赞、收藏、评论都天然依赖用户身份。
复用现有能力可以保持跨功能用户体验一致，也能避免为短视频单独维护一套令牌和用户上下文。

**Alternatives considered**:
- 匿名浏览 + 登录后互动：体验更开放，但与用户“登录必需”的最新要求不一致。
- 独立短视频账号体系：违背与主仓结构保持一致的目标。

## 决策 6：数据库采用“视频主体 + 互动状态 + 评论 + 统计快照”的最小闭环模型

**Decision**: 第一阶段数据设计聚焦 `视频主体`、`用户互动状态`、`评论` 和 `统计快照`
四类核心结构，不引入推荐召回、复杂画像、创作者中心和上传工作流。

**Rationale**: 当前需求明确聚焦观看、点赞、收藏、评论和上划切换。保持最小数据闭环可以
让 P1/P2/P3 用户故事逐层独立交付，也能控制与现有支付、消息域之间的耦合范围。

**Alternatives considered**:
- 一开始就引入复杂推荐与行为画像：成本高，超出 v1 范围。
- 只存最小评论与互动，不维护统计快照：实现更省事，但列表页会频繁聚合统计，后续扩展成本更高。

## 决策 7：测试策略采用“后端契约 + 后端集成 + iOS 核心交互”三层组合

**Decision**: 对短视频功能新增 REST 契约文档和后端接口集成测试，iOS 端补充核心交互与
状态逻辑测试；不在第一阶段扩大到全链路自动化真机回归矩阵。

**Rationale**: 该仓库已经具备 Java、BFF 和 iOS 的多层测试基础。短视频新功能风险主要
集中在接口契约、互动状态一致性和 iOS 端滑动/播放体验，采用三层组合最能覆盖高风险点。

**Alternatives considered**:
- 只做后端单元测试：无法充分覆盖接口契约和客户端状态流。
- 一开始就建设完整真机自动化矩阵：价值高，但第一阶段投入过大。
