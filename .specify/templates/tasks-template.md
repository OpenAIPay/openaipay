# 任务清单：[FEATURE NAME]

**输入**: 来自 `/specs/[###-feature-name]/` 的设计文档  
**前置条件**: `plan.md`（必需）、`spec.md`（用户故事必需）、`research.md`、`data-model.md`、`contracts/`

**测试要求**: 按宪章与实现计划补充对应的验证任务。行为变化优先使用自动化测试；如果某个用户故事采用手工验证或未新增自动化测试，必须增加显式任务，说明原因与验证步骤。

**组织方式**: 任务必须按用户故事分组，以便每个故事都能独立实现与独立测试。

## 格式：`[ID] [P?] [Story] 描述`

- **[P]**: 可并行执行（不同文件、无依赖）
- **[Story]**: 任务所属的用户故事（例如 US1、US2、US3）
- 描述中必须包含准确文件路径

## 路径约定

- **单体项目**: 仓库根目录使用 `src/`、`tests/`
- **Web 应用**: 使用 `backend/src/`、`frontend/src/`
- **移动应用**: 使用 `api/src/`、`ios/src/` 或 `android/src/`
- 以下示例默认按单体项目展示，实际生成时需根据 `plan.md` 调整

## 依赖与执行顺序

### 阶段依赖

- **准备阶段（Phase 1）**: 无依赖，可立即开始
- **基础能力阶段（Phase 2）**: 依赖 Phase 1 完成，并阻塞所有用户故事
- **用户故事阶段（Phase 3+）**: 全部依赖 Phase 2 完成
- **收尾阶段（最终阶段）**: 依赖所有目标用户故事完成

### 用户故事依赖

- **用户故事 1（P1）**: 在基础能力阶段完成后即可开始，不依赖其他故事
- **用户故事 2（P2）**: 在基础能力阶段完成后即可开始；可以与 US1 集成，但必须仍可独立验证
- **用户故事 3（P3）**: 在基础能力阶段完成后即可开始；可以与 US1/US2 集成，但必须仍可独立验证

### 每个用户故事内部的顺序

- 若包含自动化测试，自动化测试 MUST 先编写并先失败
- 模型先于服务
- 服务先于接口或交互入口
- 核心实现先于集成
- 当前故事完成并验证后，再进入下一个优先级

### 可并行机会

- 所有标记 `[P]` 的准备阶段任务可以并行
- 所有标记 `[P]` 的基础任务可以并行
- 基础能力完成后，不同用户故事可按团队容量并行展开
- 同一用户故事下标记 `[P]` 的测试任务可以并行
- 同一用户故事下标记 `[P]` 的模型任务可以并行
- 不同用户故事可由不同成员同时推进

## 实施策略

### 先做 MVP（仅用户故事 1）

1. 完成 Phase 1：准备阶段
2. 完成 Phase 2：基础能力阶段（关键阻塞）
3. 完成 Phase 3：用户故事 1
4. **停止并验证**：单独验证用户故事 1
5. 如已满足目标，可先演示或发布 MVP

### 增量交付

1. 完成准备阶段 + 基础能力阶段
2. 加入用户故事 1 → 单独验证 → 演示/发布
3. 加入用户故事 2 → 单独验证 → 演示/发布
4. 加入用户故事 3 → 单独验证 → 演示/发布
5. 每个故事都必须在不破坏前一故事价值的前提下增加新价值

### 并行协作策略

当有多位开发者时：

1. 团队共同完成准备阶段和基础能力阶段
2. 基础能力完成后：
   - 开发者 A：负责用户故事 1
   - 开发者 B：负责用户故事 2
   - 开发者 C：负责用户故事 3
3. 各故事独立完成并在后续集成

## 备注

- `[P]` 任务代表不同文件、无直接依赖，可并行执行
- `[Story]` 标签用于把任务映射回对应用户故事，确保可追溯
- 每个用户故事都应可独立完成、独立测试、独立演示
- 若包含自动化测试，需先确认测试失败，再进入实现
- 每完成一项任务或一个逻辑分组后建议提交一次
- 在每个检查点都可以停下来验证当前故事
- 避免生成模糊任务、同文件高冲突任务，以及破坏独立性的跨故事耦合
